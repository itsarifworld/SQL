The Mental Model — What Pandas Actually Is
Before you write a single line, understand what's happening under the hood
Index — The Most Misunderstood Thing in Pandas
FoundationInterview Trap
Analogy Think of a DataFrame like a spreadsheet with an invisible extra column on the left. That invisible column is the Index. It's not just a row number — it's how Pandas identifies, aligns, and retrieves data with lightning speed. Most bugs in Pandas come from not understanding the index.

Every DataFrame has an index. By default it's 0, 1, 2… but it doesn't have to be. You can set ANY column as the index. When you do operations between two DataFrames, Pandas aligns them ON THE INDEX — not by position. This causes silent bugs that are almost impossible to debug unless you understand this.

index_basics.py
# Default index — just numbers
print(df.index)           # RangeIndex(start=0, stop=8, step=1)

# Set order_id as index
df_indexed = df.set_index('order_id')
print(df_indexed.index)   # Int64Index([1001, 1002, ...])

# Reset index back to default
df_reset = df_indexed.reset_index()   # order_id becomes a column again

# THE ALIGNMENT TRAP — MUST KNOW
s1 = pd.Series([10, 20, 30], index=['a', 'b', 'c'])
s2 = pd.Series([1, 2, 3],   index=['b', 'c', 'd'])
print(s1 + s2)
# a    NaN   ← 'a' only in s1, no match in s2
# b    21.0  ← 20 + 1
# c    32.0  ← 30 + 2
# d    NaN   ← 'd' only in s2
Interview Alert — Asked at Amazon, Flipkart
  • "What happens when you add two Series with different indices?" → NaN where indices don't align
  • "When would you set_index() vs keep default?" → When you'll do many label-based lookups
  • "What does reset_index() do?" → Makes current index a column, resets to 0,1,2…
Views vs Copies — The Bug That Haunts Everyone
Classic TrapInterview Trap
Analogy A View is like a window into the original data — change the view, you change the original. A Copy is a completely new separate room. The SettingWithCopyWarning is Pandas telling you: "Hey, I'm not sure if you're modifying the original or a copy — be explicit."
views_vs_copies.py
# ❌ WRONG — This might not work as expected
electronics = df[df['category'] == 'Electronics']
electronics['amount'] = electronics['amount'] * 1.1
# SettingWithCopyWarning! You might be modifying a copy, not df

# ✅ CORRECT — Use .copy() explicitly
electronics = df[df['category'] == 'Electronics'].copy()
electronics['amount'] = electronics['amount'] * 1.1
# Now you're safely modifying a new DataFrame

# ✅ ALSO CORRECT — Use .loc for in-place modification
df.loc[df['category'] == 'Electronics', 'amount'] *= 1.1
Common Mistake That Fails Interviews Candidates often can't explain WHY SettingWithCopyWarning appears. The answer: chained indexing (df[condition][column]) is ambiguous — Pandas doesn't know if the first [] returned a view or copy. Always use .loc for assignment, or .copy() for a new DataFrame.

.loc vs .iloc — the question every interviewer asks in the first 5 minutes
.loc vs .iloc — Once and For All
Asked Every InterviewMust Know
Analogy .loc is like calling someone by their NAME — "Get me the row named 'Ananya'". .iloc is like calling them by their SEAT NUMBER — "Get me person sitting at position 0". One uses labels, one uses positions. When your index is 0,1,2,3 they look the same — but the moment your index is different, they diverge completely.
Feature.loc.iloc
UsesLabels / Index namesInteger positions (0, 1, 2...)
Slice endInclusive df.loc[0:3] → rows 0,1,2,3Exclusive df.iloc[0:3] → rows 0,1,2
Boolean mask✅ Supported✅ Supported
Best forMost use casesWhen position matters (e.g. first/last N rows)
loc_vs_iloc.py — On our Orders Dataset
# .loc — label based
df.loc[0]                          # Row with index label 0 (Ananya's first order)
df.loc[0, 'customer']              # Single value: 'Ananya'
df.loc[0:3, ['customer', 'amount']]  # Rows 0,1,2,3 (INCLUSIVE) — 4 rows
df.loc[df['amount'] > 10000]         # Boolean filter

# .iloc — position based
df.iloc[0]                           # First row by position
df.iloc[0:3]                         # Rows 0,1,2 (EXCLUSIVE) — 3 rows
df.iloc[-1]                          # Last row
df.iloc[0:4, 0:3]                    # First 4 rows, first 3 columns

# THE KEY TRAP — When index isn't 0,1,2...
df2 = df.set_index('order_id')   # index is now 1001, 1002, 1003...
df2.loc[1001]    # ✅ Gets order 1001 by label
df2.iloc[0]      # ✅ Gets first row (also order 1001) by position
df2.iloc[1001]   # ❌ IndexError — there's no position 1001 (only 8 rows)

Advanced Filtering — Beyond Basic Boolean

advanced_filtering.py
# Multiple conditions — use & | ~ (NOT 'and' 'or' 'not')
df[((df['amount'] > 10000) & (df['category'] == 'Electronics'))]

# .query() — cleaner syntax, same result
df.query("amount > 10000 and category == 'Electronics'")
df.query("city in ['Mumbai', 'Delhi']")

# .isin() — filter by list of values
df[df['city'].isin(['Mumbai', 'Delhi'])]

# .between() — range filter (inclusive both ends)
df[df['amount'].between(1000, 20000)]

# .str methods for text filtering
df[df['customer'].str.startswith('A')]   # Only 'Ananya'
df[df['city'].str.contains('ban', case=False)]  # 'Bangalore'

# Filter + select columns together
df.loc[df['delivered'] == True, ['customer', 'amount']]

# nlargest / nsmallest — top N rows
df.nlargest(3, 'amount')   # Top 3 highest orders
df.nsmallest(2, 'rating') # 2 lowest rated orders

GroupBy — The Most Tested Topic in DS Interviews
groupby + agg + transform + apply — this alone covers 40% of interview questions
GroupBy — Split, Apply, Combine
Most AskedCore
Mental Model — 3 Steps Split → Apply → Combine. Imagine physically cutting your DataFrame into piles (Split by city). Doing a calculation on each pile separately (Apply sum to each city pile). Then stacking the results back into one table (Combine). That's all groupby is.
groupby_fundamentals.py
# Basic groupby — total spend per customer
df.groupby('customer')['amount'].sum()
# customer
# Ananya    15800
# Kiran     31200
# Priya     45650
# Rohan     34500

# Multiple aggregations with .agg()
df.groupby('category')['amount'].agg(['sum', 'mean', 'count', 'max'])

# Named aggregations (clean output, interview-ready)
df.groupby('category').agg(
    total_revenue = ('amount', 'sum'),
    avg_order     = ('amount', 'mean'),
    order_count   = ('order_id', 'count'),
    avg_rating    = ('rating', 'mean')
).round(2)

# Multiple group keys
df.groupby(['city', 'category'])['amount'].sum()

# as_index=False — keeps columns flat (SQL-like output)
df.groupby('customer', as_index=False)['amount'].sum()
transform() — The Game Changer Most Candidates Don't Know
AdvancedFrequently AskedConfusing
The Key Difference groupby().sum() reduces many rows to few rows — output is SMALLER than input. transform() gives back the SAME SHAPE as original — it broadcasts the group result back to every row. This is how you add "group statistics" as a new column without losing rows.
transform_magic.py — The Most Important Distinction
# groupby().sum() — reduces rows (8 rows → 3 rows)
df.groupby('customer')['amount'].sum()
# Returns 3 rows (one per customer) — CANNOT merge back easily

# transform() — keeps all 8 rows, broadcasts group sum to each row
df['customer_total'] = df.groupby('customer')['amount'].transform('sum')
#    customer   amount  customer_total
# 0  Ananya     15000   15800   ← Ananya's total (15000 + 800)
# 1  Rohan       2500   34500   ← Rohan's total
# 2  Priya      45000   45650   ← Priya's total
# 3  Ananya       800   15800   ← Same Ananya total
# ...

# POWER USE: % contribution of each order to customer's total
df['pct_of_customer_spend'] = (
    df['amount'] / df.groupby('customer')['amount'].transform('sum') * 100
).round(1)

# Another power use: z-score within each category
df['amount_zscore'] = df.groupby('category')['amount'].transform(
    lambda x: (x - x.mean()) / x.std()
)
Classic Interview Question — Amazon, Meesho, Razorpay
  • "Add a column showing each customer's total spending alongside their individual orders" → transform('sum')
  • "Find orders where amount is above that customer's average" → filter using transform('mean')
  • "Rank orders within each category by amount" → transform + rank()
filter_with_transform.py
# Find orders above that customer's average spend
customer_avg = df.groupby('customer')['amount'].transform('mean')
df[df['amount'] > customer_avg]

# groupby filter() — keep/drop entire groups
# Keep only customers who have placed more than 1 order
df.groupby('customer').filter(lambda x: len(x) > 1)
groupby + apply() — The Swiss Army Knife
Advanced

When agg() and transform() aren't enough — when you need custom logic per group — apply() lets you pass a function that receives a mini-DataFrame for each group.

groupby_apply.py
# Top 1 order per customer by amount
df.groupby('customer').apply(lambda x: x.nlargest(1, 'amount'))

# Custom function: return orders in top 50% of that category's revenue
def top_half(group):
    threshold = group['amount'].quantile(0.5)
    return group[group['amount'] >= threshold]

df.groupby('category').apply(top_half).reset_index(drop=True)

merge() — The Full Picture
SQL Knowledge TestCore
how=What it doesSQL EquivalentWhen to use
'inner'Only matching rows in BOTHINNER JOINWhen you need guaranteed matches
'left'All left rows + matches from rightLEFT JOINKeep all left, add right info if available
'right'All right rows + matches from leftRIGHT JOINRarely used (just flip and use left)
'outer'All rows from both, NaN where no matchFULL OUTER JOINFinding what's missing in either table
'cross'Every row × every rowCROSS JOINCombinations, cartesian products
merges.py
# Second dataset for merging — customer details
customers = pd.DataFrame({
    'customer': ['Ananya', 'Rohan', 'Priya', 'Vikram'],
    'tier':     ['Gold', 'Silver', 'Gold', 'Bronze'],
    'age':      [28, 34, 25, 45]
})
# Note: 'Kiran' is in df but NOT in customers
# 'Vikram' is in customers but NOT in df

# Inner join — only customers who appear in BOTH
pd.merge(df, customers, on='customer', how='inner')
# Kiran rows DROPPED — Vikram row DROPPED

# Left join — all orders, add customer tier if available
pd.merge(df, customers, on='customer', how='left')
# Kiran rows kept, tier=NaN. Vikram row absent.

# When column names differ between tables
pd.merge(df, customers, left_on='customer', right_on='customer_name')

# Merge on multiple keys
pd.merge(df, other_df, on=['customer', 'city'], how='inner')

# Detect merge quality — ALWAYS do this in real work
pd.merge(df, customers, on='customer', how='inner', validate='many_to_one')
# validate raises error if assumptions are wrong — prevents silent bugs

# indicator=True — shows where each row came from
pd.merge(df, customers, on='customer', how='outer', indicator=True)
# Adds '_merge' column: 'left_only', 'right_only', or 'both'
# Great for finding unmatched records
Interview Scenario — "Find customers who placed an order but have no customer profile"
  • Left join df with customers, then filter where _merge == 'left_only' (using indicator=True)
  • OR: left join, filter where tier.isna()
  • This is the pandas equivalent of SQL's LEFT JOIN ... WHERE right.col IS NULL
concat() — Stacking DataFrames
Essential
merge vs concat merge is a horizontal operation — adding more columns (like SQL JOIN). concat is stacking — either adding more rows (axis=0, like SQL UNION) or adding more columns side by side (axis=1).
concat.py
df_q1 = df[df['order_date'].dt.month <= 3]
df_q2 = df[df['order_date'].dt.month > 3]

# Stack rows vertically (like UNION ALL)
df_full = pd.concat([df_q1, df_q2], ignore_index=True)
# ignore_index=True — resets index to 0,1,2... 
# Without it, index repeats: 0,1,2,3,0,1 — common bug!

# Add labels to track source
pd.concat([df_q1, df_q2], keys=['Q1', 'Q2'])
# Creates MultiIndex showing which quarter each row came from

# Stack columns horizontally (use merge instead if possible)
pd.concat([df1, df2], axis=1)
Forget ignore_index and get a bug in your interview project When you concat two DataFrames without ignore_index=True, the index from each keeps its original values. So if both go from 0–3, your result has index 0,1,2,3,0,1,2,3. Then .loc[0] returns TWO rows instead of one. Always use ignore_index=True when stacking rows.

pivot_table() — Excel Pivot in Python
EDA StapleMust Know
Analogy Think of pivot_table like a crosstab report. Rows = one category. Columns = another category. Values = what number fills the cells. Aggfunc = how to combine multiple values in one cell.
pivot_table.py
# Total amount per customer per category
df.pivot_table(
    values ='amount',
    index  ='customer',    # rows
    columns='category',    # columns
    aggfunc='sum',
    fill_value=0           # NaN → 0 where no orders exist
)
# category    Books  Electronics  Fashion
# customer
# Ananya        800        15000        0
# Kiran           0        28000     3200
# Priya         650        45000        0
# Rohan           0        34500     2500

# Multiple values and aggregations
df.pivot_table(
    values =['amount', 'rating'],
    index  ='city',
    aggfunc={'amount': 'sum', 'rating': 'mean'}
)

# margins=True — adds row/column totals (like Excel "Grand Total")
df.pivot_table(values='amount', index='city', columns='category',
              aggfunc='sum', fill_value=0, margins=True)
melt() — The Reverse of Pivot (Wide → Long)
AdvancedOften Confusing
Wide vs Long Format Wide format: each city is a separate column. Good for humans to read. Long format: city values are all in one column with a "city" label column. Good for machines and plotting. melt converts wide → long. pivot converts long → wide.
melt.py
# Wide format (what pivot_table gives us)
wide_df = df.pivot_table(values='amount', index='customer',
                           columns='category', aggfunc='sum', fill_value=0)
wide_df = wide_df.reset_index()

# melt() — convert wide back to long
long_df = wide_df.melt(
    id_vars   ='customer',             # columns to keep as-is
    value_vars=['Books', 'Electronics', 'Fashion'],  # columns to melt
    var_name  ='category',            # name for the "label" column
    value_name='total_spend'          # name for the "value" column
)
# customer   category     total_spend
# Ananya     Books        800
# Ananya     Electronics  15000
# Ananya     Fashion      0
# ...

# Long format is what seaborn/plotly prefers for plotting!

apply(), map(), applymap() — Know the Difference
One question, three tools — interviewers love testing which one you reach for
apply vs map vs applymap (now map in newer pandas)
Commonly TestedEasy to Confuse
MethodWorks onReceivesBest for
.map()Series onlyOne element at a timeReplacing values, simple transforms
.apply()Series OR DataFrameElement (Series) or row/col (DataFrame)Complex custom logic
.applymap() / .map()DataFrame element-wiseOne element at a timeFormatting every cell
apply_map.py
# map() — on Series — replace values or apply function
df['category'].map({'Electronics': 'Tech', 'Fashion': 'Style', 'Books': 'Read'})
df['amount'].map(lambda x: 'High' if x > 10000 else 'Low')

# apply() on Series — same as map() but more powerful
df['amount'].apply(lambda x: round(x * 1.18, 2))  # amount with 18% GST

# apply() on DataFrame — row-wise (axis=1) or column-wise (axis=0)
df.apply(lambda row: row['amount'] * (1 - row['discount_pct']/100), axis=1)
# axis=1 → function receives each ROW as a Series
# axis=0 → function receives each COLUMN as a Series

# PERFORMANCE TIP — avoid apply when vectorization works
# ❌ Slow:
df['final_amount'] = df.apply(lambda r: r['amount'] * (1 - r['discount_pct']/100), axis=1)
# ✅ Fast (vectorized):
df['final_amount'] = df['amount'] * (1 - df['discount_pct'] / 100)

# np.where — fastest for conditional columns
df['order_size'] = np.where(df['amount'] > 10000, 'Large', 'Small')

# np.select — multiple conditions (best for 3+ conditions)
conditions = [
    df['amount'] > 30000,
    df['amount'].between(10000, 30000),
    df['amount'] < 10000
]
choices = ['Premium', 'Standard', 'Budget']
df['segment'] = np.select(conditions, choices)
Golden Rule — Hierarchy of Speed Vectorized ops (df['a'] + df['b']) → np.where/np.select → .map() → .apply(axis=1) → Python for loop (never). Always think: can I do this without apply? If yes, do that. Interviewers at product companies ask about this explicitly.

Time Series in Pandas
dt accessor, resample, rolling — used in almost every real DS project
.dt accessor + resample() + rolling()
AdvancedProject Staple
time_series.py
# .dt accessor — extract date parts
df['year']    = df['order_date'].dt.year
df['month']   = df['order_date'].dt.month
df['day']     = df['order_date'].dt.day
df['weekday'] = df['order_date'].dt.day_name()  # 'Monday', etc.
df['quarter'] = df['order_date'].dt.quarter
df['week']    = df['order_date'].dt.isocalendar().week

# Set date as index for time series operations
ts_df = df.set_index('order_date')

# resample() — aggregate by time period (like groupby for dates)
ts_df['amount'].resample('M').sum()   # Monthly total revenue
ts_df['amount'].resample('W').mean()  # Weekly average order
ts_df['amount'].resample('Q').agg(['sum', 'count'])  # Quarterly
# Freq codes: 'D'=day, 'W'=week, 'M'=month, 'Q'=quarter, 'Y'=year

# rolling() — moving window calculations
monthly = ts_df['amount'].resample('M').sum()
monthly.rolling(window=3).mean()   # 3-month moving average
monthly.rolling(window=3).std()    # 3-month rolling std

# shift() and diff() — lag features for ML
monthly.shift(1)         # Previous month's value (lag 1)
monthly.diff(1)          # Change from previous month
monthly.pct_change()     # % change month-over-month

Advanced Operations
cut, qcut, method chaining, pipe, assign — what senior candidates know
cut() and qcut() — Binning Data
Feature EngAdvanced
cut vs qcut cut = you define the bin EDGES (fixed ranges). qcut = you define the number of QUANTILE bins (equal frequency). cut is for "amount 0-10000 = budget", qcut is for "bottom 25% = low spenders".
binning.py
# pd.cut() — fixed boundary bins
df['spend_tier'] = pd.cut(
    df['amount'],
    bins   =[0, 5000, 20000, 50000],
    labels =['Budget', 'Mid', 'Premium'],
    right  =True   # right-inclusive: (0,5000]
)

# pd.qcut() — quantile-based bins (equal frequency)
df['spend_quartile'] = pd.qcut(
    df['amount'],
    q     =4,
    labels=['Q1', 'Q2', 'Q3', 'Q4']
)

# Use in groupby right away
df.groupby('spend_tier', observed=True)['rating'].mean()
Method Chaining + pipe() — Write Code Like a Pro
AdvancedImpressive in Interviews
Why Method Chaining? Instead of creating temp variables at every step (df1, df2, df3...), you chain operations. Your code reads like a story: "take df, filter it, add a column, group it, sort it." One expression. Zero temp variables. Senior data scientists write this way.
method_chaining.py
# ❌ Amateur style — temp variables everywhere
df1 = df[df['delivered'] == True]
df2 = df1.copy()
df2['final_amount'] = df2['amount'] * (1 - df2['discount_pct']/100)
df3 = df2.groupby('category')['final_amount'].sum().reset_index()
df4 = df3.sort_values('final_amount', ascending=False)

# ✅ Pro style — method chaining with .assign()
result = (
    df
    .loc[df['delivered'] == True]
    .assign(final_amount = lambda x: x['amount'] * (1 - x['discount_pct']/100))
    .groupby('category')['final_amount']
    .sum()
    .reset_index()
    .sort_values('final_amount', ascending=False)
    .rename(columns={'final_amount': 'Revenue After Discount'})
)

# .pipe() — for custom functions in a chain
def add_gst(df, rate=0.18):
    return df.assign(gst_amount=df['amount'] * rate)

def flag_premium(df, threshold=20000):
    return df.assign(is_premium=df['amount'] > threshold)

result = (
    df
    .pipe(add_gst, rate=0.18)
    .pipe(flag_premium, threshold=20000)
)
Missing Values — The Complete Playbook
Always AskedReal Data
missing_values.py
# Detect — df has NaN in 'rating' column
df.isnull().sum()          # Count NaN per column
df.isnull().sum() / len(df) # % missing per column
df[df['rating'].isnull()]  # Rows with missing rating

# Drop
df.dropna()                     # Drop ANY row with NaN
df.dropna(subset=['rating'])    # Drop only if 'rating' is NaN
df.dropna(thresh=5)             # Keep rows with at least 5 non-NaN values

# Fill
df['rating'].fillna(df['rating'].mean())       # Fill with mean
df['rating'].fillna(df['rating'].median())     # Fill with median
df['rating'].fillna(method='ffill')           # Forward fill
df['rating'].fillna(method='bfill')           # Backward fill

# Smart fill — fill with group mean (better than global mean)
df['rating'] = df.groupby('category')['rating'].transform(
    lambda x: x.fillna(x.mean())
)

# INTERVIEW QUESTION: Which strategy when?
# Numerical, random missing → mean/median
# Time series → ffill/bfill
# Categorical → mode or 'Unknown'
# Many missing (>40%) → consider dropping the column

Performance & Memory Optimization
What separates a junior from a senior in one question
dtype optimization + Categorical type
Senior LevelFrequently Asked
performance.py
# Check memory usage
df.info(memory_usage='deep')
df.memory_usage(deep=True)

# Optimize int columns — downcast to smallest int type
df['amount'] = pd.to_numeric(df['amount'], downcast='integer')
# int64 (8 bytes) → int16 (2 bytes) — 4x memory saving!

# Categorical dtype — for low-cardinality string columns
# 'category' only has 3 unique values — wastes memory as object dtype
df['category'] = df['category'].astype('category')
df['city']     = df['city'].astype('category')
# Stores strings as integer codes internally — massive memory savings
# Also speeds up groupby and sort operations!

# Rule of thumb: if unique values < 50% of total rows → use category
for col in df.select_dtypes(include='object').columns:
    if df[col].nunique() / len(df) < 0.5:
        df[col] = df[col].astype('category')

# Read large CSVs efficiently
df = pd.read_csv('large_file.csv',
    usecols=['customer', 'amount', 'category'],  # only needed cols
    dtype={'category': 'category', 'amount': 'int32'},
    parse_dates=['order_date'],
    chunksize=10000   # for very large files — process in chunks
)
