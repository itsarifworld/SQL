The #1 Thing Nobody Teaches — SQL Execution Order
Why your WHERE can't use aliases. Why HAVING exists. All explained here.
How SQL Actually Processes Your Query
Critical Foundation
AnalogyYou write SQL top to bottom. SQL executes it in a completely different order. It's like writing an essay — you write the title first but the reader understands the conclusion only after reading the body. Knowing this order explains every confusing error you've ever seen.
1
FROM / JOINBuild the full table — bring all data together from all tables
Tables assembled here
2
WHEREFilter individual rows — before any grouping happens
Row-level filter
3
GROUP BYBucket rows into groups by specified columns
Grouping here
4
HAVINGFilter groups — like WHERE but for groups after aggregation
Group-level filter
5
SELECTChoose which columns to show, apply expressions, aliases
Columns chosen here
6
ORDER BYSort the final result — can use aliases from SELECT!
Aliases usable here
7
LIMIT / OFFSETTake only N rows from the sorted result
Final cut
Why This Matters in Interviews
  • "Why can't I use a SELECT alias in WHERE?" → WHERE runs before SELECT, alias doesn't exist yet
  • "Why does HAVING exist if we have WHERE?" → WHERE can't filter aggregates (SUM, COUNT) because GROUP BY hasn't happened yet
  • "Can ORDER BY use aliases?" → YES — ORDER BY runs after SELECT, so aliases exist
  • "Why can't GROUP BY use window function results?" → Window functions run after GROUP BY in logical order

select_basics.sql
-- Basic structure
SELECT column1, column2, expression AS alias
FROM   table_name
WHERE  condition
ORDER BY column1 DESC
LIMIT  10;

-- All operators you must know
WHERE amount = 5000          -- equal
WHERE amount != 5000         -- not equal (also <>)
WHERE amount > 1000 AND amount < 10000
WHERE amount BETWEEN 1000 AND 10000   -- inclusive both ends
WHERE city IN ('Mumbai', 'Delhi', 'Bangalore')
WHERE city NOT IN ('Chennai')
WHERE name LIKE 'A%'         -- starts with A (% = any chars)
WHERE name LIKE '%kumar'     -- ends with kumar
WHERE name LIKE '%an%'       -- contains 'an' anywhere
WHERE name LIKE 'An_nya'     -- _ matches exactly one char
WHERE discount_pct IS NULL
WHERE discount_pct IS NOT NULL

Aggregations — GROUP BY, HAVING
COUNT, SUM, AVG, MIN, MAX — and why HAVING is not just "WHERE for groups"
Aggregate Functions Deep Dive
Most AskedCore
Mental ModelGROUP BY physically separates your table into piles (one per unique group value). Then aggregate functions run on each pile independently. HAVING is the filter that runs AFTER the piles are made — it can see COUNT(*), SUM(), etc. WHERE runs BEFORE piles are made — it cannot.
aggregations.sql
-- COUNT variations — most misunderstood in interviews
SELECT
  COUNT(*)            AS total_rows,       -- counts everything including NULLs
  COUNT(discount_pct) AS non_null_discounts, -- ignores NULLs
  COUNT(DISTINCT user_id) AS unique_buyers, -- distinct count
  SUM(amount)         AS total_revenue,
  AVG(amount)         AS avg_order_value,
  MIN(amount)         AS min_order,
  MAX(amount)         AS max_order,
  ROUND(AVG(amount),2) AS rounded_aov
FROM orders
WHERE status = 'completed';  -- filters BEFORE grouping

-- GROUP BY with multiple columns
SELECT
  u.city,
  o.status,
  COUNT(*) AS order_count,
  SUM(o.amount) AS revenue
FROM  orders o
JOIN  users  u ON o.user_id = u.user_id
GROUP BY u.city, o.status
ORDER BY revenue DESC;

-- HAVING — filter on aggregated result
SELECT
  user_id,
  COUNT(*) AS order_count,
  SUM(amount) AS total_spend
FROM  orders
GROUP BY user_id
HAVING COUNT(*) >= 3           -- only users with 3+ orders
   AND SUM(amount) > 10000;    -- AND spent over 10000

-- HAVING vs WHERE — clear comparison
SELECT city, COUNT(*) AS users
FROM users
WHERE  plan = 'pro'        -- ✅ filter rows BEFORE grouping
GROUP BY city
HAVING COUNT(*) > 5;       -- ✅ filter groups AFTER grouping
-- ❌ Wrong: WHERE COUNT(*) > 5  → error, aggregate in WHERE not allowed
Classic Interview Traps
  • COUNT(*) vs COUNT(col): COUNT(*) counts all rows; COUNT(col) skips NULLs — asked at every company
  • Every column in SELECT must be in GROUP BY OR be an aggregate function — exception: primary key
  • AVG ignores NULLs — SUM/COUNT also ignore NULLs — this can make AVG misleading
  • "Find cities with more than 100 users" → GROUP BY city HAVING COUNT(*) > 100
Business Use Case — Flipkart/Amazon StyleFind customers who made at least 2 purchases in Q1 2024 and spent over ₹5000 total. Flag them as "high value" for a loyalty campaign. → GROUP BY user_id HAVING COUNT(*) >= 2 AND SUM(amount) > 5000

CASE WHEN — SQL's if/else
Segmentation, conditional aggregation, pivot — all powered by CASE WHEN
CASE WHEN — Every Pattern You Need
EverywhereProduct Analytics
case_when.sql
-- Simple CASE (like switch statement)
SELECT
  user_id,
  plan,
  CASE plan
    WHEN 'free'  THEN 'Freemium'
    WHEN 'basic' THEN 'Starter'
    WHEN 'pro'   THEN 'Premium'
    ELSE 'Unknown'
  END AS plan_label
FROM users;

-- Searched CASE (most common — each WHEN has full condition)
SELECT
  order_id, amount,
  CASE
    WHEN amount >= 10000 THEN 'Premium'
    WHEN amount >= 3000  THEN 'Standard'
    WHEN amount >= 500   THEN 'Budget'
    ELSE                       'Micro'
  END AS order_tier
FROM orders;

-- CONDITIONAL AGGREGATION — the killer pattern for interviews
-- "How many orders per status per month in ONE query?"
SELECT
  DATE_TRUNC('month', ordered_at) AS month,
  COUNT(*) AS total_orders,
  COUNT(CASE WHEN status = 'completed' THEN 1 END) AS completed,
  COUNT(CASE WHEN status = 'refunded'  THEN 1 END) AS refunded,
  COUNT(CASE WHEN status = 'pending'   THEN 1 END) AS pending,
  SUM(CASE WHEN status = 'completed'  THEN amount ELSE 0 END) AS completed_revenue,
  ROUND(100.0 * COUNT(CASE WHEN status='refunded' THEN 1 END) / COUNT(*), 2) AS refund_rate
FROM orders
GROUP BY 1   -- GROUP BY 1 = GROUP BY first column in SELECT
ORDER BY 1;

-- CASE in ORDER BY — custom sort logic
SELECT * FROM orders
ORDER BY
  CASE status
    WHEN 'pending'   THEN 1   -- pending first
    WHEN 'completed' THEN 2
    WHEN 'refunded'  THEN 3
  END;

-- PIVOT with CASE WHEN — show categories as columns
SELECT
  u.city,
  COUNT(CASE WHEN u.plan = 'free'  THEN 1 END) AS free_users,
  COUNT(CASE WHEN u.plan = 'basic' THEN 1 END) AS basic_users,
  COUNT(CASE WHEN u.plan = 'pro'   THEN 1 END) AS pro_users
FROM users u
GROUP BY u.city;
Pro Tip — Conditional Aggregation is Pivot in SQLSUM(CASE WHEN condition THEN amount ELSE 0 END) is how you "pivot" rows into columns. This is asked in 70% of senior DA interviews at product companies. COUNT(CASE WHEN condition THEN 1 END) — note: ELSE NULL by default, so NULLs are skipped by COUNT, giving you conditional count.

JOINs — Complete Guide
INNER, LEFT, RIGHT, FULL, CROSS, SELF, Anti-Join — every type with business context
Visual Guide to All JOIN Types
Most Tested
⬛🔲
INNER JOIN
Only matching rows in BOTH tables. Most restrictive.
🔲⬛
LEFT JOIN
All left rows + matches from right. NULL where no match.
⬛🔲
RIGHT JOIN
All right rows + matches from left. Just flip LEFT JOIN.
🔲🔲
FULL OUTER
All rows from both. NULL where no match on either side.
🔲✖️🔲
CROSS JOIN
Every row × every row. N×M rows. No ON clause needed.
🔄
SELF JOIN
Table joined to itself. Employee→Manager hierarchy.
joins_complete.sql
-- INNER JOIN — only users who have placed orders
SELECT u.name, u.city, o.amount, o.status
FROM   users  u
INNER JOIN orders o ON u.user_id = o.user_id;

-- LEFT JOIN — ALL users, orders if they exist (NULL if not)
SELECT u.name, u.city, o.amount
FROM  users  u
LEFT JOIN orders o ON u.user_id = o.user_id;
-- Users who never ordered: o.amount IS NULL

-- ANTI-JOIN — Users who NEVER placed an order
-- Pattern: LEFT JOIN + WHERE right side IS NULL
SELECT u.name, u.city
FROM  users  u
LEFT JOIN orders o ON u.user_id = o.user_id
WHERE o.order_id IS NULL;     -- the magic line

-- SELF JOIN — find users from the same city
SELECT a.name AS user1, b.name AS user2, a.city
FROM  users a
JOIN  users b ON a.city = b.city AND a.user_id < b.user_id;
-- a.user_id < b.user_id prevents (A,B) and (B,A) duplicates

-- FULL OUTER JOIN — find unmatched on BOTH sides
SELECT
  COALESCE(u.user_id, o.user_id) AS uid,
  u.name, o.order_id,
  CASE
    WHEN u.user_id IS NULL THEN 'Order without user'
    WHEN o.order_id IS NULL THEN 'User without order'
    ELSE 'Matched'
  END AS match_status
FROM      users  u
FULL OUTER JOIN orders o ON u.user_id = o.user_id;

-- JOIN on multiple conditions
SELECT * FROM orders o
JOIN courses c ON o.course_id = c.course_id
JOIN users   u ON o.user_id   = u.user_id
WHERE c.category = 'Data Science' AND u.plan = 'pro';
Interview Questions — JOINs at FAANG/Product Companies
  • "Find users who signed up but never made a purchase" → Anti-JOIN (LEFT + WHERE IS NULL)
  • "What's the difference between LEFT JOIN and INNER JOIN?" → LEFT keeps all left rows
  • "If I JOIN two tables and get more rows than the left table, what happened?" → Duplicate keys in right table causing fan-out
  • "Find customers who bought course A but NOT course B" → Anti-JOIN pattern
FAN-OUT BUG — The Hidden JOIN TrapIf the right table has multiple matching rows per key, JOIN duplicates the left table rows. SUM() after a duplicating JOIN gives WRONG results. Always check: is my join key unique in the right table? If not, deduplicate first or use a subquery.

Subqueries — Queries Inside Queries
Scalar, correlated, EXISTS — and when to use CTE instead
All Subquery Types with Examples
Medium InterviewsAdvanced
subqueries.sql
-- 1. SCALAR SUBQUERY — returns exactly one value
SELECT
  user_id, amount,
  amount - (SELECT AVG(amount) FROM orders) AS diff_from_avg
FROM orders;

-- 2. IN SUBQUERY — filter where value exists in sub-result
SELECT name, city
FROM  users
WHERE user_id IN (
  SELECT DISTINCT user_id
  FROM orders
  WHERE amount > 5000
);

-- 3. CORRELATED SUBQUERY — references outer query (runs per row)
-- "Find orders above that user's own average"
SELECT o1.user_id, o1.amount
FROM  orders o1
WHERE o1.amount > (
  SELECT AVG(o2.amount)
  FROM  orders o2
  WHERE o2.user_id = o1.user_id  -- references outer query!
);

-- 4. EXISTS — true/false existence check (faster than IN for large sets)
SELECT u.name
FROM  users u
WHERE EXISTS (
  SELECT 1
  FROM orders o
  WHERE o.user_id = u.user_id
    AND o.status = 'completed'
);

-- 5. FROM subquery — treat result as table
SELECT city, avg_spend
FROM (
  SELECT u.city, AVG(o.amount) AS avg_spend
  FROM  users  u
  JOIN  orders o ON u.user_id = o.user_id
  GROUP BY u.city
) AS city_stats
WHERE avg_spend > 3000   -- can now filter on aggregated result
ORDER BY avg_spend DESC;
EXISTS vs IN — Which is Faster?EXISTS short-circuits — stops as soon as it finds one match. IN evaluates ALL values. For large subquery results: EXISTS is faster. For small lists: IN is cleaner. In most modern databases the optimizer handles this, but interviewers still ask — answer: EXISTS is semantically correct for "does any row match."

CTEs — Common Table Expressions
WITH clauses, multiple CTEs, recursive CTEs — write readable SQL like a senior analyst
CTEs: The Right Way to Structure Complex Queries
Senior LevelReadability
AnalogyA CTE is like giving a name to a sticky note. Instead of copy-pasting the same subquery three times in your query, you write it once with a name (WITH ...) and refer to that name. It doesn't improve performance over subqueries in most databases — but it makes your code 10x more readable and interviewers LOVE it.
ctes.sql — Building a complex analytics query step by step
-- Single CTE
WITH high_value_users AS (
  SELECT user_id, SUM(amount) AS total_spend
  FROM  orders
  WHERE status = 'completed'
  GROUP BY user_id
  HAVING SUM(amount) > 10000
)
SELECT u.name, u.city, h.total_spend
FROM  users       u
JOIN  high_value_users h ON u.user_id = h.user_id
ORDER BY h.total_spend DESC;

-- MULTIPLE CTEs — chained like a pipeline
WITH

-- Step 1: Get monthly order stats
monthly_orders AS (
  SELECT
    DATE_TRUNC('month', ordered_at) AS month,
    user_id,
    COUNT(*) AS orders_in_month,
    SUM(amount) AS monthly_spend
  FROM orders
  WHERE status = 'completed'
  GROUP BY 1, 2
),

-- Step 2: Classify users by activity
user_segments AS (
  SELECT
    user_id,
    AVG(orders_in_month) AS avg_monthly_orders,
    SUM(monthly_spend)   AS total_spend,
    CASE
      WHEN AVG(orders_in_month) >= 2 THEN 'Power User'
      WHEN AVG(orders_in_month) >= 1 THEN 'Regular'
      ELSE                               'Occasional'
    END AS segment
  FROM monthly_orders    -- references first CTE!
  GROUP BY user_id
),

-- Step 3: Count by segment
segment_summary AS (
  SELECT
    segment,
    COUNT(*)           AS user_count,
    AVG(total_spend)   AS avg_ltv
  FROM user_segments   -- references second CTE!
  GROUP BY segment
)

-- Final SELECT
SELECT * FROM segment_summary ORDER BY avg_ltv DESC;
recursive_cte.sql — Hierarchy / Date Series
-- RECURSIVE CTE — Generate date series (PostgreSQL/BigQuery)
WITH RECURSIVE date_series AS (
  SELECT '2024-01-01'::DATE AS dt          -- anchor
  UNION ALL
  SELECT dt + INTERVAL '1 day'
  FROM date_series
  WHERE dt < '2024-03-31'                  -- termination
)
SELECT dt FROM date_series;
-- Use case: LEFT JOIN this to daily orders to fill 0s on days with no orders

-- RECURSIVE CTE — Employee-Manager hierarchy
WITH RECURSIVE org_tree AS (
  SELECT employee_id, name, manager_id, 0 AS level
  FROM  employees
  WHERE manager_id IS NULL      -- start at root (CEO)
  UNION ALL
  SELECT e.employee_id, e.name, e.manager_id, t.level + 1
  FROM  employees e
  JOIN  org_tree   t ON e.manager_id = t.employee_id
)
SELECT * FROM org_tree ORDER BY level;

Window Functions — The Most Powerful SQL Feature
ROW_NUMBER, RANK, LAG, LEAD, SUM OVER, NTILE — this alone covers 50% of hard interviews
The Core Mental Model — OVER() Clause
CriticalFAANG Level
Window vs GROUP BY — The Fundamental DifferenceGROUP BY collapses many rows into fewer rows (one per group). Window functions keep ALL rows but ADD a computed column based on a "window" of rows. The window slides along your data. This is why they're called window functions — you're looking through a sliding window.
window_anatomy.sql — Understand this, everything else follows
-- Full syntax of a window function
function_name() OVER (
  PARTITION BY column1          -- divide into groups (optional)
  ORDER BY     column2          -- sort within each group (required for ranking/running)
  ROWS BETWEEN start AND end   -- define window frame (optional)
)

-- OVER() with nothing = entire table as one window
SELECT user_id, amount, AVG(amount) OVER() AS global_avg
FROM orders;  -- every row shows the SAME global avg alongside its own amount

-- PARTITION BY = GROUP BY but keeps all rows
SELECT
  user_id, amount,
  SUM(amount) OVER(PARTITION BY user_id) AS user_total,
  amount / SUM(amount) OVER(PARTITION BY user_id) AS pct_of_user_total
FROM orders;
Ranking Functions — ROW_NUMBER, RANK, DENSE_RANK, NTILE
Top-N ProblemsFAANG Staple
FunctionBehavior on tiesUse case
ROW_NUMBER()Always unique — tie broken arbitrarilyDe-deduplication, getting exactly 1 row per group
RANK()Ties get same rank, GAPS after tieCompetition ranking (1,1,3,4)
DENSE_RANK()Ties get same rank, NO GAPSPercentile ranking (1,1,2,3)
NTILE(n)Divides into n equal bucketsQuartiles, deciles, top 10%
ranking.sql
-- All four on same data to see the difference
SELECT
  user_id, amount,
  ROW_NUMBER()   OVER(ORDER BY amount DESC) AS row_num,
  RANK()         OVER(ORDER BY amount DESC) AS rank_num,
  DENSE_RANK()   OVER(ORDER BY amount DESC) AS dense_rank_num,
  NTILE(4)       OVER(ORDER BY amount DESC) AS quartile
FROM orders;
-- If amounts are: 500, 500, 300, 200, 100
-- row_num:     1, 2, 3, 4, 5
-- rank:        1, 1, 3, 4, 5   ← gap after tie
-- dense_rank:  1, 1, 2, 3, 4   ← no gap
-- ntile(4):    1, 1, 2, 3, 4   ← four buckets

-- THE CLASSIC INTERVIEW QUESTION:
-- "Find the top 2 orders by amount for each user"
WITH ranked AS (
  SELECT
    user_id, order_id, amount,
    ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY amount DESC) AS rn
  FROM orders
)
SELECT user_id, order_id, amount
FROM  ranked
WHERE rn <= 2;         -- ← the filter on window function result

-- WHY ROW_NUMBER not RANK here?
-- If user has two orders both ₹5000, RANK gives both rank 1, so WHERE rn<=2
-- might return 3 rows (two rank-1 + one rank-3). ROW_NUMBER gives exactly 2.

-- Top spender per city
WITH city_ranked AS (
  SELECT
    u.name, u.city, SUM(o.amount) AS total,
    RANK() OVER(PARTITION BY u.city ORDER BY SUM(o.amount) DESC) AS rnk
  FROM  orders o JOIN users u ON o.user_id = u.user_id
  GROUP BY u.name, u.city
)
SELECT name, city, total
FROM  city_ranked
WHERE rnk = 1;
LAG & LEAD — Compare Rows Across Time
Time SeriesMust Know
AnalogyLAG looks BACKWARD — "what was the value N rows ago?" LEAD looks FORWARD — "what is the value N rows ahead?" This is how you compare this month to last month, or find users who upgraded their plan.
lag_lead.sql
-- Month-over-month revenue change
WITH monthly AS (
  SELECT
    DATE_TRUNC('month', ordered_at) AS month,
    SUM(amount) AS revenue
  FROM orders
  WHERE status = 'completed'
  GROUP BY 1
)
SELECT
  month,
  revenue,
  LAG(revenue) OVER(ORDER BY month) AS prev_month_revenue,
  revenue - LAG(revenue) OVER(ORDER BY month) AS mom_change,
  ROUND(100.0 *
    (revenue - LAG(revenue) OVER(ORDER BY month))
    / NULLIF(LAG(revenue) OVER(ORDER BY month), 0),
  2) AS mom_pct_change
FROM monthly;

-- LAG with PARTITION BY — per user comparison
SELECT
  user_id, ordered_at, amount,
  LAG(amount, 1, 0) OVER(PARTITION BY user_id ORDER BY ordered_at) AS prev_order,
  -- LAG(col, offset, default) — 1 row back, default 0 if no prior row
  amount - LAG(amount,1,0) OVER(PARTITION BY user_id ORDER BY ordered_at) AS spend_change
FROM orders;

-- LEAD — find next event for each user (churn prediction)
SELECT
  user_id, event_type, event_at,
  LEAD(event_type) OVER(PARTITION BY user_id ORDER BY event_at) AS next_event,
  LEAD(event_at)   OVER(PARTITION BY user_id ORDER BY event_at) AS next_event_at
FROM events;
Running Totals, Moving Averages, Cumulative Sums
Analytics CoreProduct
running_totals.sql
-- ROWS BETWEEN — the frame clause
-- UNBOUNDED PRECEDING = from very first row in window
-- CURRENT ROW        = this row
-- N PRECEDING        = N rows before current
-- UNBOUNDED FOLLOWING = to very last row in window

SELECT
  ordered_at::DATE AS dt,
  SUM(amount) AS daily_revenue,

  -- Running total from day 1 to current day
  SUM(SUM(amount)) OVER(
    ORDER BY ordered_at::DATE
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_revenue,

  -- 7-day moving average
  AVG(SUM(amount)) OVER(
    ORDER BY ordered_at::DATE
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS ma7_revenue,

  -- 30-day moving average
  AVG(SUM(amount)) OVER(
    ORDER BY ordered_at::DATE
    ROWS BETWEEN 29 PRECEDING AND CURRENT ROW
  ) AS ma30_revenue

FROM  orders
WHERE status = 'completed'
GROUP BY ordered_at::DATE
ORDER BY dt;

-- Cumulative user count over time (growth tracking)
WITH daily_signups AS (
  SELECT joined_at::DATE AS dt, COUNT(*) AS new_users
  FROM  users
  GROUP BY 1
)
SELECT
  dt, new_users,
  SUM(new_users) OVER(ORDER BY dt) AS total_users
FROM daily_signups;
FAANG Interview: "Compute 7-day rolling average of daily active users"
  • Step 1: Get daily DAU with GROUP BY date
  • Step 2: AVG(COUNT(*)) OVER(ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
  • Step 3: Note: need to fill 0 for days with no activity using date series + LEFT JOIN
FIRST_VALUE, LAST_VALUE, NTH_VALUE, PERCENT_RANK
Advanced
advanced_window.sql
-- FIRST_VALUE / LAST_VALUE — access first/last row in window
SELECT
  user_id, ordered_at, amount,
  FIRST_VALUE(amount) OVER(
    PARTITION BY user_id ORDER BY ordered_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS first_order_amount,
  LAST_VALUE(amount) OVER(
    PARTITION BY user_id ORDER BY ordered_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING  -- IMPORTANT!
  ) AS latest_order_amount
FROM orders;
-- CRITICAL: LAST_VALUE default frame is CURRENT ROW, not end.
-- Always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING

-- PERCENT_RANK — percentile position (0 to 1)
SELECT
  user_id, amount,
  PERCENT_RANK() OVER(ORDER BY amount) AS pct_rank,
  CUME_DIST()    OVER(ORDER BY amount) AS cumulative_dist
FROM orders;
-- PERCENT_RANK: (rank-1)/(total-1) — 0 to 1
-- CUME_DIST: rows with val <= current / total — proportion

date_functions.sql — PostgreSQL/BigQuery syntax (most common in interviews)
-- Extracting parts
EXTRACT(YEAR FROM  ordered_at)   -- 2024
EXTRACT(MONTH FROM ordered_at)   -- 1-12
EXTRACT(DOW FROM   ordered_at)   -- 0=Sunday ... 6=Saturday
EXTRACT(HOUR FROM  ordered_at)   -- 0-23
DATE_PART('week', ordered_at)    -- ISO week number

-- Truncate to period (great for GROUP BY)
DATE_TRUNC('day',     ordered_at) -- 2024-01-15 00:00:00
DATE_TRUNC('week',    ordered_at) -- 2024-01-15 (Monday of that week)
DATE_TRUNC('month',   ordered_at) -- 2024-01-01 00:00:00
DATE_TRUNC('quarter', ordered_at) -- 2024-01-01 (Q1 start)
DATE_TRUNC('year',    ordered_at) -- 2024-01-01 00:00:00

-- Date arithmetic
ordered_at + INTERVAL '7 days'      -- add 7 days
ordered_at - INTERVAL '1 month'     -- subtract 1 month
ordered_at::DATE - joined_at::DATE    -- days between (integer)
CURRENT_DATE                         -- today's date
NOW()                                -- current timestamp
CURRENT_DATE - INTERVAL '30 days'  -- 30 days ago

-- Days between two dates
DATE_PART('day', ordered_at - joined_at::TIMESTAMP) AS days_to_order
-- Or: ordered_at::DATE - joined_at  (integer subtraction)

-- Convert string to date
TO_DATE('2024-01-15', 'YYYY-MM-DD')
'2024-01-15'::DATE   -- PostgreSQL cast
CAST('2024-01-15' AS DATE)  -- standard SQL

-- TYPICAL INTERVIEW PATTERN
SELECT
  DATE_TRUNC('month', ordered_at) AS month,
  COUNT(*) AS orders,
  SUM(amount) AS revenue
FROM orders
WHERE ordered_at >= DATE_TRUNC('year', CURRENT_DATE)  -- this year
GROUP BY 1 ORDER BY 1;

NULL Handling — The Silent Data Killer
COALESCE, NULLIF, ISNULL — NULLs in JOINs, aggregations, and comparisons
Everything About NULLs
Trap Alert
Mental Model — NULL is "Unknown"NULL ≠ 0. NULL ≠ empty string. NULL means "we don't know this value." Any comparison with NULL returns NULL (unknown), not TRUE or FALSE. 5 = NULL → NULL. NULL = NULL → NULL. This is why = doesn't work and IS NULL is needed.
null_handling.sql
-- COALESCE — return first non-NULL value
COALESCE(discount_pct, 0)            -- if NULL, use 0
COALESCE(promo_code, promo_name, 'NONE') -- first non-null of three

-- NULLIF — return NULL if two values are equal
NULLIF(denominator, 0)              -- prevents division by zero!
-- 100 / NULLIF(0, 0) → NULL (safe)  vs  100/0 → ERROR (crash)
amount / NULLIF(prev_amount, 0)     -- always use this in % calculations

-- NULLs in aggregates (surprising behavior)
COUNT(*)           -- includes NULL rows
COUNT(rating)      -- EXCLUDES NULL ratings
AVG(rating)        -- ignores NULLs — can be misleading
SUM(rating)        -- ignores NULLs — SUM of all non-NULL
MAX(rating)        -- ignores NULLs

-- NULL in ORDER BY
ORDER BY rating DESC NULLS LAST   -- NULLs appear at end
ORDER BY rating ASC NULLS FIRST   -- NULLs appear at start

-- NULL safe division — the pattern every analyst must know
SELECT
  month,
  revenue,
  prev_revenue,
  ROUND(
    100.0 * (revenue - prev_revenue) / NULLIF(prev_revenue, 0),
  2) AS growth_pct  -- NULL if prev_revenue is 0, not an error
FROM monthly_revenue;

-- Counting NULLs explicitly
COUNT(*) - COUNT(rating) AS null_count  -- elegant way to count NULLs

-- NULL in JOIN — NULLs never match each other
-- If join key has NULLs, those rows are excluded from INNER JOIN
-- Always check for NULL join keys in real data!
SELECT * FROM orders WHERE user_id IS NOT NULL;  -- clean before joining

Pattern 1: De-duplication — Get One Row Per Group
Asked Everywhere
dedup.sql
-- Most recent order per user (de-duplication)
-- Method 1: ROW_NUMBER (most readable, preferred)
WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY ordered_at DESC) AS rn
  FROM orders
)
SELECT * FROM ranked WHERE rn = 1;

-- Method 2: Correlated subquery
SELECT * FROM orders o
WHERE ordered_at = (
  SELECT MAX(ordered_at) FROM orders
  WHERE user_id = o.user_id
);

-- Deduplicate exact duplicate rows
WITH deduped AS (
  SELECT *, ROW_NUMBER() OVER(PARTITION BY user_id, ordered_at, amount ORDER BY order_id) AS rn
  FROM orders
)
SELECT * FROM deduped WHERE rn = 1;
Pattern 2: Gaps & Islands — Consecutive Events
HardFAANG
What is Gaps & Islands?"Find users who were active for 3 consecutive days." "Find streaks." "Find periods of continuous activity." The classic approach: row_number minus a date sequence creates groups (islands) for consecutive events.
gaps_islands.sql
-- Find consecutive login streaks per user
WITH

-- Step 1: Get distinct login dates per user
login_days AS (
  SELECT DISTINCT user_id, event_at::DATE AS login_date
  FROM events
  WHERE event_type = 'login'
),

-- Step 2: Assign row number per user, subtract from date → group key
grouped AS (
  SELECT
    user_id, login_date,
    login_date - (ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY login_date))::INT AS grp
    -- Consecutive dates get the SAME grp value! This is the trick.
  FROM login_days
),

-- Step 3: Group by user + grp → each consecutive run is one island
streaks AS (
  SELECT
    user_id,
    MIN(login_date) AS streak_start,
    MAX(login_date) AS streak_end,
    COUNT(*) AS streak_length
  FROM  grouped
  GROUP BY user_id, grp
)

SELECT * FROM streaks
WHERE streak_length >= 3        -- streaks of 3+ days
ORDER BY streak_length DESC;   -- longest first

Cohort Analysis
Group users by signup month, track their behavior over time
What is a Cohort?Users who joined in January 2024 are "the January cohort." Cohort analysis tracks what percentage of that group came back in Month 2, Month 3, etc. This reveals whether product improvements are actually improving retention.
cohort_analysis.sql — Full cohort retention table
WITH

-- Step 1: Assign each user their cohort (signup month)
user_cohorts AS (
  SELECT
    user_id,
    DATE_TRUNC('month', joined_at) AS cohort_month
  FROM users
),

-- Step 2: Get each user's activity months
user_activity AS (
  SELECT DISTINCT
    e.user_id,
    DATE_TRUNC('month', e.event_at) AS activity_month
  FROM events e
  WHERE e.event_type IN ('login','lesson_start','purchase')
),

-- Step 3: Join and compute month number (0=joined, 1=month after, etc.)
cohort_data AS (
  SELECT
    uc.cohort_month,
    ua.user_id,
    EXTRACT(MONTH FROM AGE(ua.activity_month, uc.cohort_month)) AS month_number
  FROM  user_cohorts  uc
  JOIN  user_activity ua ON uc.user_id = ua.user_id
  WHERE ua.activity_month >= uc.cohort_month
),

-- Step 4: Cohort sizes (how many users in each cohort)
cohort_sizes AS (
  SELECT cohort_month, COUNT(DISTINCT user_id) AS cohort_size
  FROM  user_cohorts
  GROUP BY cohort_month
)

-- Step 5: Final retention table
SELECT
  cd.cohort_month,
  cs.cohort_size,
  cd.month_number,
  COUNT(DISTINCT cd.user_id) AS active_users,
  ROUND(100.0 * COUNT(DISTINCT cd.user_id) / cs.cohort_size, 1) AS retention_pct
FROM  cohort_data  cd
JOIN  cohort_sizes cs ON cd.cohort_month = cs.cohort_month
GROUP BY cd.cohort_month, cs.cohort_size, cd.month_number
ORDER BY cd.cohort_month, cd.month_number;

Retention Analysis — Day 1, Day 7, Day 30
What % of users come back after 1, 7, 30 days
retention.sql — Classic N-day retention
WITH

first_activity AS (
  SELECT user_id, MIN(event_at::DATE) AS first_date
  FROM  events
  GROUP BY user_id
),

-- Check if user was active on day 1, 7, 30 after first activity
retention AS (
  SELECT
    f.user_id,
    f.first_date,
    MAX(CASE WHEN e.event_at::DATE = f.first_date + 1  THEN 1 ELSE 0 END) AS day1,
    MAX(CASE WHEN e.event_at::DATE = f.first_date + 7  THEN 1 ELSE 0 END) AS day7,
    MAX(CASE WHEN e.event_at::DATE = f.first_date + 30 THEN 1 ELSE 0 END) AS day30
  FROM  first_activity f
  LEFT JOIN events e ON f.user_id = e.user_id
  GROUP BY f.user_id, f.first_date
)

SELECT
  first_date,
  COUNT(*) AS new_users,
  ROUND(100.0*SUM(day1) /COUNT(*),1) AS d1_retention,
  ROUND(100.0*SUM(day7) /COUNT(*),1) AS d7_retention,
  ROUND(100.0*SUM(day30)/COUNT(*),1) AS d30_retention
FROM retention
GROUP BY first_date
ORDER BY first_date;

Funnel Analysis
Conversion at each step of user journey
funnel.sql — Signup → Lesson → Purchase funnel
WITH

funnel_steps AS (
  SELECT
    user_id,
    MAX(CASE WHEN event_type = 'signup'           THEN 1 ELSE 0 END) AS did_signup,
    MAX(CASE WHEN event_type = 'lesson_start'     THEN 1 ELSE 0 END) AS did_lesson,
    MAX(CASE WHEN event_type = 'lesson_complete'  THEN 1 ELSE 0 END) AS did_complete,
    MAX(CASE WHEN event_type = 'purchase'         THEN 1 ELSE 0 END) AS did_purchase
  FROM events
  GROUP BY user_id
)

SELECT
  COUNT(*)                     AS total_users,
  SUM(did_signup)              AS step1_signup,
  SUM(did_lesson)              AS step2_lesson,
  SUM(did_complete)            AS step3_complete,
  SUM(did_purchase)            AS step4_purchase,
  ROUND(100.0*SUM(did_lesson) /NULLIF(SUM(did_signup),0),1) AS step1_to_2_cvr,
  ROUND(100.0*SUM(did_complete)/NULLIF(SUM(did_lesson),0),1) AS step2_to_3_cvr,
  ROUND(100.0*SUM(did_purchase)/NULLIF(SUM(did_complete),0),1) AS step3_to_4_cvr,
  ROUND(100.0*SUM(did_purchase)/NULLIF(SUM(did_signup),0),1)  AS overall_cvr
FROM funnel_steps;

DAU, WAU, MAU — Engagement Metrics
Daily/Weekly/Monthly Active Users — the heartbeat of product analytics
dau_mau.sql
-- Daily Active Users (DAU)
SELECT
  event_at::DATE AS dt,
  COUNT(DISTINCT user_id) AS dau
FROM  events
GROUP BY 1 ORDER BY 1;

-- Monthly Active Users (MAU) — unique users in last 30 days
SELECT
  DATE_TRUNC('month', event_at) AS month,
  COUNT(DISTINCT user_id) AS mau
FROM  events
GROUP BY 1 ORDER BY 1;

-- DAU/MAU ratio — stickiness metric (higher = better engagement)
WITH
dau AS (
  SELECT event_at::DATE AS dt, COUNT(DISTINCT user_id) AS daily_users
  FROM  events GROUP BY 1
),
mau AS (
  SELECT DATE_TRUNC('month', event_at)::DATE AS month,
         COUNT(DISTINCT user_id) AS monthly_users
  FROM  events GROUP BY 1
)
SELECT
  d.dt,
  d.daily_users AS dau,
  m.monthly_users AS mau,
  ROUND(100.0 * d.daily_users / NULLIF(m.monthly_users,0), 1) AS stickiness_pct
FROM  dau d
JOIN  mau m ON DATE_TRUNC('month', d.dt) = m.month
ORDER BY d.dt;

-- New vs Returning users per day
WITH first_seen AS (
  SELECT user_id, MIN(event_at::DATE) AS first_date FROM events GROUP BY user_id
)
SELECT
  e.event_at::DATE AS dt,
  COUNT(DISTINCT CASE WHEN e.event_at::DATE = f.first_date THEN e.user_id END) AS new_users,
  COUNT(DISTINCT CASE WHEN e.event_at::DATE > f.first_date  THEN e.user_id END) AS returning_users
FROM  events e
JOIN  first_seen f ON e.user_id = f.user_id
GROUP BY 1 ORDER BY 1;

Problem 1: User Activity Histogram
HardMeta/Facebook
Problem
  • Write a SQL query to find the number of users who were active for at least 1 day, 2 days, 3 days, and so on, during Q1 2024. Arrange the result as: active_days | user_count | pct_of_total
Step-by-Step Solution
1Count distinct active days per user in Q1 2024
2Group by that count to get distribution
3Add cumulative count for "at least N days" interpretation
histogram.sql
WITH
user_active_days AS (
  SELECT
    user_id,
    COUNT(DISTINCT event_at::DATE) AS active_days
  FROM  events
  WHERE event_at >= '2024-01-01' AND event_at < '2024-04-01'
  GROUP BY user_id
),
distribution AS (
  SELECT
    active_days,
    COUNT(*) AS user_count
  FROM  user_active_days
  GROUP BY active_days
)
SELECT
  active_days,
  user_count,
  ROUND(100.0*user_count/SUM(user_count)OVER(),2) AS pct,
  -- "at least N days" = sum of all users with days >= current
  SUM(user_count) OVER(ORDER BY active_days DESC
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS at_least_n_users
FROM  distribution
ORDER BY active_days;
Problem 2: Median Calculation (No MEDIAN function in most DBs)
HardAmazon/Flipkart
median.sql — Two approaches
-- Method 1: PERCENTILE_CONT (PostgreSQL, BigQuery)
SELECT
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY amount) AS median_amount,
  PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY amount) AS p25,
  PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY amount) AS p75
FROM orders;

-- Method 2: ROW_NUMBER trick (works everywhere)
WITH ranked AS (
  SELECT
    amount,
    ROW_NUMBER() OVER(ORDER BY amount) AS rn,
    COUNT(*) OVER() AS total
  FROM orders
)
SELECT AVG(amount) AS median
FROM  ranked
WHERE rn IN (FLOOR((total+1)/2.0), CEIL((total+1)/2.0));
-- Handles both even and odd number of rows correctly
Problem 3: Friend Recommendation (Graph Problem)
HardMeta/LinkedIn
Problem
  • Given a friendships table (user1_id, user2_id), find pairs of users who have at least 2 mutual friends but are NOT already friends with each other.
mutual_friends.sql
-- friendships table: user1_id, user2_id (undirected — store both ways)
WITH

-- Normalize: ensure (A,B) and (B,A) both exist
all_friends AS (
  SELECT user1_id AS user_id, user2_id AS friend_id FROM friendships
  UNION ALL
  SELECT user2_id, user1_id FROM friendships
),

-- Find mutual friends: A-C and B-C → A and B share friend C
mutual AS (
  SELECT
    a.user_id AS person_a,
    b.user_id AS person_b,
    COUNT(*) AS mutual_friend_count
  FROM  all_friends a
  JOIN  all_friends b ON a.friend_id = b.friend_id
                      AND a.user_id < b.user_id   -- avoid duplicates
                      AND a.user_id != b.user_id   -- not same person
  GROUP BY a.user_id, b.user_id
  HAVING COUNT(*) >= 2         -- at least 2 mutual friends
)

-- Exclude pairs who are ALREADY friends
SELECT m.person_a, m.person_b, m.mutual_friend_count
FROM  mutual m
WHERE NOT EXISTS (
  SELECT 1 FROM all_friends af
  WHERE af.user_id = m.person_a AND af.friend_id = m.person_b
)
ORDER BY mutual_friend_count DESC;
Problem 4: Longest Streak of Daily Purchases
HardFlipkart/Amazon
longest_streak.sql — Gaps & Islands applied
WITH

-- Step 1: One row per user per purchase day
purchase_days AS (
  SELECT DISTINCT user_id, ordered_at::DATE AS purchase_date
  FROM  orders
  WHERE status = 'completed'
),

-- Step 2: The gaps & islands trick
grouped AS (
  SELECT
    user_id, purchase_date,
    purchase_date - CAST(
      ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY purchase_date)
    AS INT) AS grp
  FROM purchase_days
),

-- Step 3: Each streak = same grp value per user
streaks AS (
  SELECT
    user_id, grp,
    MIN(purchase_date) AS streak_start,
    MAX(purchase_date) AS streak_end,
    COUNT(*) AS streak_days
  FROM  grouped
  GROUP BY user_id, grp
),

-- Step 4: Longest streak per user
max_streaks AS (
  SELECT
    user_id,
    MAX(streak_days) AS max_streak
  FROM  streaks
  GROUP BY user_id
)

SELECT
  ms.user_id,
  u.name,
  ms.max_streak,
  s.streak_start,
  s.streak_end
FROM  max_streaks ms
JOIN  streaks s ON ms.user_id = s.user_id AND ms.max_streak = s.streak_days
JOIN  users   u ON ms.user_id = u.user_id
ORDER BY ms.max_streak DESC
LIMIT 10;

Interview Cheat Sheet + Optimization Tips
Quick reference for the day before your interview

Execution Order

  • 1 — FROM / JOIN
  • 2 — WHERE
  • 3 — GROUP BY
  • 4 — HAVING
  • 5 — SELECT
  • 6 — ORDER BY
  • 7 — LIMIT

JOIN Quick Reference

  • INNER — only matches
  • LEFT — all left + matches
  • Anti-JOIN — LEFT + WHERE IS NULL
  • FULL OUTER — all both sides
  • CROSS — all combinations
  • SELF — table joins itself

Window Function Reference

  • ROW_NUMBER — unique, no ties
  • RANK — gaps after ties
  • DENSE_RANK — no gaps
  • LAG(col,n,default) — look back
  • LEAD(col,n,default) — look ahead
  • SUM OVER — running total
  • NTILE(n) — bucket into n groups

NULL Functions

  • COALESCE(a,b,c) — first non-null
  • NULLIF(a,b) — null if a=b
  • IS NULL / IS NOT NULL
  • NULLS LAST in ORDER BY
  • COUNT(*) includes nulls
  • COUNT(col) excludes nulls

Date Functions

  • DATE_TRUNC('month',dt)
  • EXTRACT(YEAR FROM dt)
  • CURRENT_DATE — today
  • dt + INTERVAL '7 days'
  • date1 - date2 → integer days
  • TO_DATE(str,'YYYY-MM-DD')

Optimization Tips

  • Filter early — WHERE before JOIN
  • Avoid SELECT * in production
  • Use EXISTS not IN for large sets
  • EXPLAIN ANALYZE to see plan
  • Partition large tables by date
  • Index on join keys and WHERE cols
  • LIMIT in subqueries when possible

Product SQL Patterns

  • Cohort = group by signup month
  • Retention = % back after N days
  • Funnel = MAX(CASE WHEN step)
  • DAU = COUNT(DISTINCT user_id)
  • Stickiness = DAU/MAU × 100
  • LTV = SUM(revenue) per user

Top-N Pattern (Must Know)

  • WITH ranked AS (SELECT *, ROW_NUMBER() OVER(PARTITION BY grp ORDER BY col DESC) rn FROM t)
  • SELECT * FROM ranked WHERE rn <= N
  • Use RANK for ties allowed
  • Use ROW_NUMBER for exactly N
🧠 The 5 Interview Questions That Appear at EVERY Company
  • "Find the Nth highest salary/revenue" → DENSE_RANK() OVER(ORDER BY col DESC) = N
  • "Users who never did X" → LEFT JOIN + WHERE right.id IS NULL (Anti-Join)
  • "Month-over-month growth" → LAG(revenue) OVER(ORDER BY month) then % change
  • "Top N per group" → ROW_NUMBER() OVER(PARTITION BY group ORDER BY col DESC) + WHERE rn <= N
  • "Running total/cumulative sum" → SUM(col) OVER(ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
